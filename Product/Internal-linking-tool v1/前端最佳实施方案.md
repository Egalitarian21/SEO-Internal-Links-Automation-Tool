# 电商博客内链工作台 v1：前端最佳实施方案

## 1. 方案目标与原则

本方案服务于 `Internal-linking-tool v1` 的前端落地，目标是在 40 天内由单人完成可上线版本，并满足以下核心要求：

- 支撑完整业务链路：项目管理 -> URL 导入 -> Wiki 知识库 -> 内链推荐 -> 审核发布 -> 任务监控。
- 面向 SEO 实施团队，强调高效率批量操作与可解释结果展示。
- 与后端异步任务模型兼容，确保长耗时流程可观测、可重试。
- 低成本优先，前端部署在 Vercel，尽量减少自建基础设施。

执行原则：

- 功能优先于炫技：先保证端到端可用，再做交互精修。
- 组件化与可复用：统一数据表格、状态标签、检查清单、审核卡片。
- 强约束可视化：规则违规、三层相关性评分、发布前检查必须显式呈现。
- 默认可扩展：为后续“多客户规则配置”“健康度仪表盘”预留扩展位。

## 2. 技术选型（最佳组合）

## 2.1 栈选择

- 框架：`Next.js 15 (App Router + TypeScript)`  
- UI：`shadcn/ui + Tailwind CSS + Lucide`
- 状态管理：`TanStack Query + Zustand`
- 表单与校验：`react-hook-form + zod`
- 表格：`TanStack Table`
- 鉴权：`Auth.js v5`（内部团队登录）
- 观测：`Sentry (前端)` + 自定义任务日志视图

## 2.2 选型理由

- `Next.js 15` 同时支持 SSR/CSR 与路由分层，适合后台工作台。
- `TanStack Query` 自带缓存、重试、轮询、乐观更新，天然适配“异步任务+状态轮询”。
- `shadcn/ui` 开箱快且可控，单人开发效率高，长期维护成本低。

## 3. 信息架构与路由设计

建议采用“全局壳 + 项目上下文”结构：

- `/(auth)/login`
- `/dashboard`
- `/projects`
- `/projects/new`
- `/projects/[projectId]/import`
- `/projects/[projectId]/articles`
- `/projects/[projectId]/cards`
- `/projects/[projectId]/recommend`
- `/projects/[projectId]/review`
- `/projects/[projectId]/reverse-links`
- `/projects/[projectId]/tasks`
- `/settings`（预留）

导航层级：

- 一级：Dashboard / 项目管理 / 项目工作台 / 任务监控
- 二级（项目工作台）：URL 导入 / Wiki 知识库 / 内链推荐 / 审核发布 / 反向内链

## 4. 前端模块拆分

## 4.1 页面级模块

- `ProjectListPage`：项目列表、状态统计、快捷进入工作台。
- `ImportPage`：URL 批量导入、导入进度、失败项重试。
- `CardsPage`：Wiki 卡片列表、搜索过滤、卡片编辑。
- `RecommendPage`：选文章触发推荐，展示建议及规范合规状态。
- `ReviewPage`：Approve/Reject/Edit、批量审核、发布前检查、执行发布。
- `TasksPage`：异步任务队列、进度条、失败详情、重试入口。

## 4.2 领域组件（必须复用）

- `SuggestionReviewCard`：展示锚文本上下文、目标页面、评分、违规信息。
- `RuleViolationBadgeGroup`：规则维度标签（阻断/警告）。
- `RelevanceScoreTriple`：三层相关性可视化（anchor-target / paragraph-target / continuity）。
- `PrePublishChecklistPanel`：发布前检查清单与阻断原因。
- `AsyncTaskTimeline`：任务状态流（queued/running/success/failed）。
- `SmartTable`：统一分页、筛选、排序、批量选择。

## 4.3 数据层与 API 客户端

- `lib/api/client.ts`：统一请求封装、鉴权头、错误标准化。
- `lib/api/endpoints.ts`：按 PRD 的 API 分组定义。
- `lib/query/keys.ts`：统一 queryKey，避免缓存冲突。
- `lib/query/mutations.ts`：导入、推荐、审核、执行发布等变更操作。

## 5. 关键交互流（落地级）

## 5.1 URL 导入与异步监控

1. 用户上传 URL 列表触发 `POST /api/projects/:id/import`。  
2. 前端写入任务中心并开始轮询 `import-status`。  
3. 实时显示：总数、已完成、失败数、失败原因。  
4. 支持“一键重试失败 URL”。  

## 5.2 内链推荐与审核

1. 用户在文章列表选择目标文章，触发推荐。  
2. 推荐结果以卡片流显示，默认按“规则风险 > 相关性分数”排序。  
3. 每条建议可 `Approve / Edit / Reject`。  
4. 批量审核时显示风险汇总，避免误批。  

## 5.3 发布前检查与执行

1. 审核完成后触发 `pre-publish-check`。  
2. 阻断项（首段有链接、H 标签有链接、通用锚文本）红色高亮。  
3. 全部通过后启用“确认发布”。  
4. 发布成功后记录 `publish_job_id` 并提供回滚入口。  

## 6. 状态管理与性能策略

- 使用 Query 分层缓存：项目维度、文章维度、建议维度。
- 对列表页启用增量加载与虚拟滚动，避免长列表卡顿。
- 推荐结果采用“先骨架后流式填充”，提升体感速度。
- 所有长任务采用指数退避轮询（2s -> 5s -> 10s，上限 30s）。
- 统一错误边界：接口失败、鉴权过期、任务超时、网络断开。

## 7. 前端开发里程碑（40 天对齐）

## M1（Day 1-8）：前端骨架

- 项目初始化、路由壳、Auth.js、基础 Layout、项目列表页。

## M2（Day 9-18）：导入与知识库视图

- URL 导入页、任务监控、Wiki 卡片列表与搜索过滤、卡片编辑弹窗。

## M3（Day 19-28）：推荐与规则可视化

- 推荐页、审核卡片、三层评分组件、规则违规标签、批量操作。

## M4（Day 29-36）：发布链路打通

- 发布前检查面板、执行发布、回滚入口、状态追踪。

## M5（Day 37-40）：体验打磨

- 空状态/错误态优化、键盘快捷操作、筛选性能优化、可观测性完善。

## 8. 风险与应对

- 异步任务接口不稳定：前端做任务态机与幂等重试，不依赖单次响应成功。
- 推荐结果字段变化频繁：对接口响应做 `zod` 解码与版本兼容。
- 批量审核误操作：增加二次确认与撤销窗口（短时 Undo）。
- 大项目数据量膨胀：提前预留分页、服务端筛选与虚拟列表。

## 9. 验收标准（前端）

- 核心流程可在 UI 端闭环完成，无需命令行介入。
- 推荐审核页可清晰看到：锚文本上下文、目标类型、相关性、规则状态。
- 发布前检查可明确区分“警告”与“阻断”。
- 长任务有可视化进度，失败可追踪可重试。
- 页面在常见项目规模下交互流畅（列表/筛选/批量操作无明显卡顿）。
