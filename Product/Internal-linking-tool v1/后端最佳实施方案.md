# 电商博客内链工作台 v1：后端最佳实施方案

## 1. 方案目标与约束

本方案用于落地 `Internal-linking-tool v1` 的后端实现，围绕 PRD 的关键约束设计：

- 单人全栈、40 天交付、免费起步。
- 核心能力不是“接口 CRUD”，而是“异步 AI 流水线 + 规则引擎 + 可回滚发布”。
- 必须稳定支撑：导入抓取 -> Wiki 卡片构建 -> 推荐排序 -> 人工审核 -> Shopify 发布。

目标结果：

- 后端服务可稳定处理分钟级异步任务与秒级审核操作。
- 推荐结果具备可解释性（三层相关性 + 规则违规明细）。
- 发布流程具备强安全边界（检查清单阻断 + before_html 快照 + 回滚）。

## 2. 最佳技术方案（推荐唯一主方案）

## 2.1 技术栈

- Web 框架：`FastAPI + Uvicorn`
- 任务队列：`Celery + Redis`（Railway 托管 Redis）
- 数据层：`Supabase Postgres + pgvector`
- ORM/数据访问：`SQLAlchemy 2.x + Alembic`
- LLM 编排：`LangChain`（调用私有 OpenAI-compatible 接口）
- 模型：
  - 生成与排序：`Qwen3-32B-Instruct`
  - 向量化：`bge-large-en-v1.5`
- 内容处理：`readability-lxml + trafilatura + bs4`
- 发布对接：`Shopify Admin API`

## 2.2 方案优势

- Python 生态完整覆盖爬取、清洗、LLM 编排、NLP、向量检索。
- Celery 明确隔离耗时任务，避免 API 阻塞，适配 Railway 持久进程。
- Supabase 一体化 SQL + 向量，降低架构复杂度与维护成本。

## 3. 后端总体架构

```txt
FastAPI API Layer
  -> Application Services
     -> Task Dispatcher (Celery)
        -> Worker Pipelines
           -> LLM/Embedding Clients
           -> Rules Engine
           -> HTML Inserter
           -> Shopify Publisher
  -> Postgres(Supabase)+pgvector
  -> Redis(Celery broker/result)
```

关键原则：

- 同步接口只做“快速响应 + 任务入队”。
- 异步 worker 负责“重计算与外部依赖调用”。
- 所有核心环节写入状态与日志，确保可追踪、可重试。

## 4. 模块设计（按职责分层）

## 4.1 API 路由层（routers）

- `imports.py`：导入 URL、查看导入进度。
- `drafts.py`：文章列表与详情。
- `cards.py`：Wiki 卡片查询与编辑。
- `suggestions.py`：推荐触发、列表筛选、反向推荐。
- `review.py`：审核、批量审核。
- `publish.py`：检查清单、执行发布、回滚。

## 4.2 服务层（services）

- `import_service`：URL 去重、任务拆分、导入态机。
- `crawl_service`：抓取、超时控制、失败重试。
- `clean_service`：正文提取、段落切分、标题/H 标签解析。
- `card_service`：Wiki 卡片生成、JSON 结构校验、入库。
- `recommend_service`：锚文本提取、候选召回、LLM 精排。
- `rules_service`：位置硬约束、锚文本约束、数量分布校验。
- `publish_service`：预检、HTML 插入、Shopify 写入、回滚。

## 4.3 基础设施层（infra）

- `llm_client.py`：统一模型调用、重试、超时、JSON 模式。
- `embedding_client.py`：批量 embedding 与降级处理。
- `vector_repo.py`：pgvector 检索封装（wiki_cards/link_targets）。
- `shopify_client.py`：请求签名、限流控制、错误映射。
- `task_bus.py`：Celery task ID 关联业务 ID。

## 5. 核心流程实施细则

## 5.1 冷启动建库（导入 -> 抓取 -> 卡片）

1. 接收 URL 列表，写入 `imported_blog_urls`（status=`queued`）。  
2. Celery 并发抓取，提取正文/Meta/H 标签，写入 `blog_drafts`。  
3. 对博客文章触发 Wiki 卡片生成，严格校验 JSON 字段完整性。  
4. 写入 `wiki_cards` 与向量字段；产品/集合页写入 `link_targets`。  

异常策略：

- 单 URL 失败不阻断全局；记录错误并允许重试。
- LLM 输出不合法时自动重试（最多 2 次），仍失败则标记人工复核。

## 5.2 内链推荐（高价值主链路）

1. LLM 从正文提取锚文本候选（必须命中原文，至少 2 词）。  
2. 规则引擎执行硬过滤（首段、H 标签、冒号前、重复目标等）。  
3. 对每个锚文本分别召回博客候选与产品/集合候选。  
4. LLM 对合并候选执行三层相关性评分与精排。  
5. 规则引擎后校验：数量达标、分布均衡、URL 规范化。  
6. 写入 `anchor_suggestions`，附带 `relevance_details` 与 `rule_violations`。  

## 5.3 审核与发布

1. 审核通过的建议汇总生成“发布草案”。  
2. 执行检查清单，阻断项未过禁止发布。  
3. 生成 `after_html` 前保存 `before_html` 快照。  
4. 调 Shopify API 更新文章，写入 `shopify_publish_jobs`。  
5. 发布失败支持按 job 回滚。  

## 6. 数据库实施重点

## 6.1 索引建议

- `blog_drafts(project_id, article_id)`
- `wiki_cards(project_id, source_article_id, page_type)`
- `link_targets(project_id, page_type)`
- `anchor_suggestions(source_article_id, status, created_at desc)`
- `review_tasks(suggestion_id, reviewed_at desc)`
- 向量索引：`ivfflat` 或 `hnsw`（依据 Supabase 版本）

## 6.2 数据一致性原则

- 使用事务包裹“建议生成批次入库”，避免部分写入。
- 审核与执行操作做幂等校验（同 suggestion 不可重复执行）。
- 发布与回滚采用 job 状态机：`queued -> checking -> publishing -> success/failed -> rolled_back`。

## 7. 规则引擎实现建议

引擎结构建议“可组合规则”：

- `position_rules.py`：P-01 ~ P-06
- `anchor_rules.py`：A-01 ~ A-07
- `quantity_rules.py`：产品/博客数量约束
- `url_rules.py`：Shopify canonical 规范化
- `checklist_rules.py`：发布前检查清单

执行模式：

- 推荐前：硬约束直接过滤。
- 推荐后：输出违规明细，不满足阻断条件时禁止执行发布。
- 对外统一返回：
  - `severity`: `blocker | warning`
  - `rule_id`
  - `message`

## 8. API 设计增强建议（在 PRD 基础上补足）

- 所有异步触发接口返回 `task_id`。
- 增加统一任务查询接口：`GET /api/tasks/:id`。
- 推荐列表支持分页+多维筛选：
  - `status`
  - `page_type`
  - `has_rule_violations`
  - `min_relevance_score`
- 批量执行接口返回“成功/失败明细”，避免黑盒结果。

## 9. 安全、稳定性与可观测

- Shopify Token 使用 KMS 或环境变量加密存储，日志中绝不输出明文。
- 全链路 request_id 贯穿 API -> task -> LLM -> DB 写入。
- 对 LLM/Embedding 接口增加超时、熔断和退避重试。
- 任务级 metrics：
  - 导入成功率
  - 卡片生成成功率
  - 推荐平均耗时
  - 发布成功率
  - 规则违规分布

## 10. 后端 40 天里程碑落地

## M1（Day 1-8）：基础设施

- FastAPI 骨架、数据库迁移、项目/导入基础 API、Shopify 连通性验证。

## M2（Day 9-18）：数据管线

- 抓取与清洗、Wiki 卡片生成、pgvector 写入、任务状态可见。

## M3（Day 19-28）：推荐引擎

- 锚文本提取、双通道召回、三层评分、规则引擎前后校验。

## M4（Day 29-36）：审核发布

- 审核流程、检查清单、HTML 插入、发布与回滚闭环。

## M5（Day 37-40）：稳定性打磨

- 限流与重试策略、边界 case、日志完善、团队试运行。

## 11. 验收标准（后端）

- 单篇推荐在目标性能内完成（常规 < 30s）。
- 推荐结果可追溯：每条均可看到评分与规则判断依据。
- 发布流程有阻断边界，不合规建议不可执行。
- 发布失败可一键回滚且数据一致。
- 核心异步任务均可重试、可定位失败原因。
