# 无标题文档

### 一、 技术栈选型 (Technology Stack)

考虑到工程实施效率与 AI 集成便利性，选定以下技术栈：

*   **后端语言：** Python 3.10+
    
*   **Web 框架：** FastAPI 
    
*   **向量数据库：** PostgreSQL + pgvector 
    
*   **核心库：**
    
*   `sentence-transformers`: 用于本地化语义向量化。
    
*   `scikit-learn`: 用于进行 K-Means 聚类分析。
    
*   `BeautifulSoup4`: 用于解析与注入 HTML 锚文本。
    
*   **UI 框架：** React + Ant Design 。
    

---

### 二、 核心功能模块与逻辑流

#### 1. 数据对齐与预处理 (Ingestion)

系统接入客户站点库与内部编辑组的 CMS 接口，将文章分为 `source_type: client` 和 `source_type: internal`。

#### 2. 语义集群构建 (Clustering)

利用本地模型将文章标题与正文转化为 768 维向量，存储至 `pgvector`。

*   **逻辑：** 计算 `internal` 文章与 `client` 文章的余弦相似度。
    
*   **目标：** 自动将新写的定制博客归类到现有的客户频道或 Pillar Page（支柱页面）下。
    

#### 3. AI 驱动的内链策略提取 (AI API Integration)

通过接入模型的 API，仅在“决策”环节产生 token 消耗，而非全量扫描。

*   **输入：** 定制文章正文 + 候选支柱页面 URL/标题 + 关键词库。
    
*   **输出：** 推荐的锚文本位置（Index）、最匹配的 URL、链接上下文的语义分值。

### 三、 核心代码实现流程

```python
from sentence_transformers import SentenceTransformer
import psycopg2
# 初始化本地开源模型，无需 API 费用
model = SentenceTransformer('all-MiniLM-L6-v2')
def vectorize_and_store(article_id, content):
    vector = model.encode(content).tolist()
    # 存入 PostgreSQL pgvector
    cursor.execute("UPDATE blog_repository SET embedding = %s WHERE id = %s", (vector, article_id))
```

```sql
-- 寻找与当前定制博客语义最相关的 3 篇客户存量博客作为潜在支柱
SELECT id, title, url, 
       (embedding <=> '%s') AS distance 
FROM blog_repository 
WHERE source_type = 'client' 
ORDER BY distance LIMIT 3;
```

import openai
def get_internal_link_suggestion(current_article, candidate_urls):

    Task: Identify the most natural anchor text in the provided article for the target URLs.

    Candidates: {candidate_urls}
    Output: JSON only [{{ "anchor": "...", "url": "...", "context_sentence": "..." }}]

### 四、 UI/UX 效率设计 (实施工程师导向)

为提高实施效率，UI 界面采用\*\*“批量建议审阅流”\*\*而非单篇操作：

*   **双栏对比视图：** 左侧显示定制博客，右侧高亮显示 AI 建议的锚文本位置及指向的客户博客 URL。
    
*   **一键应用机制：** 工程师勾选确认无误的链接，系统通过正则表达式自动在 HTML 中插入 `<a>` 标签并同步至 CMS 数据库。
    
*   **冲突预警：** 若同一段落已存在内链，界面自动标红，防止过度优化（Over-optimization）。

### 五、 日志系统与异常处理 (Stability)

为了防止系统在高并发或 API 异常时崩溃，集成 `Loguru` 进行多级日志追踪。

```python
from loguru import logger
# 配置日志分片与保留策略
logger.add("logs/seo_automation_{time}.log", rotation="500 MB", retention="10 days", level="INFO")
def process_internal_links(task_id):
    try:
        logger.info(f"Task {task_id} started: Fetching data from DB.")
        # 执行内链逻辑...
    except Exception as e:
        logger.error(f"Task {task_id} failed. Error: {str(e)}")
        # 写入数据库错误状态，触发系统前端弹窗警告
        update_task_status(task_id, "FAILED", error_msg=str(e))
```

#### 日志追踪关键点：

1.  **API Latency:** 记录 AI 接口响应时间。
    
2.  **DB Transaction:** 记录 HTML 注入前后的数据备份状态。
    
3.  **Crawl Errors:** 若客户网站无法访问，记录 404/500 状态码。

### 六、 实施成本分析 (Maintenance)

*   **服务器成本：** 仅需 1 台 4 核 8G 云服务器即可运行 Python 后端与 PostgreSQL。
    
*   **API 成本：** 通过本地向量化过滤，仅 5% 的高价值页面需要调用 AI API 进行锚文本精修，极大地降低了 Token 费用。
    
*   **人工成本：** 自动化集群技术将原有的手动查词、手动找链接的时间从平均 30 分钟/篇降至 2 分钟/篇。
